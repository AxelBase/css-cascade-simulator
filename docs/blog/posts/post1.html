Understanding CSS Specificity

Understanding CSS Specificity: The Core Algorithm

Published: December 3, 2025

CSS specificity is the mechanism browsers use to decide which style rule wins when multiple rules target the same element. It is not about which rule comes last in the file, but about mathematical weight. Every selector has a specificity score made up of three numbers, often written as a, b, c.

How Specificity Is Calculated

The first number counts how many ID selectors are in the rule. An ID like header gives a very high score because IDs are meant to be unique. The second number counts class selectors, attribute selectors, and pseudo-classes like hover or focus. These are common but less powerful than IDs. The third number counts element selectors like div, p, or h1, and pseudo-elements like before or after.

When two rules compete, the browser compares these three numbers from left to right. The rule with the higher value in the first differing position wins. If all three numbers are equal, only then does the order in the file matter, with later rules winning.

Why This Matters in Practice

A single ID selector will always beat one hundred class selectors. This is why a rule like header wins over .header-text .active .button even if the class rule appears later. Understanding this prevents confusion when styles don’t apply as expected.

Common Misconceptions

Many developers believe that the last rule always wins. This is only true when specificity is tied. Another common mistake is thinking that more selectors mean higher specificity. Ten classes do not beat one ID. The type of selector matters far more than the quantity.

By mastering specificity, you write predictable CSS that is easier to maintain and debug. This simulator helps you see exactly how the browser makes its decisions, removing guesswork from styling.

Mastering specificity is the foundation of professional CSS architecture.
