Source Order and the Cascade

Why Source Order Only Matters When Specificity Ties

Published: December 3, 2025

One of the most common misunderstandings in CSS is believing that the last rule in a stylesheet always wins. This is not true. Source order is only used as a final tiebreaker when two or more rules have exactly the same specificity score.

The Real Decision Process

When multiple rules apply to the same element, the browser follows a strict order. First, it compares specificity scores. The rule with the higher specificity wins immediately, regardless of where it appears in the code. Only if the specificity values are identical does the browser look at the order in which the rules were defined.

This means a lower-specificity rule that comes later will lose to a higher-specificity rule that comes earlier. This behavior surprises many developers who expect the last rule to win no matter what.

Practical Examples

Two rules with the same class selector will follow source order. If both target .button and have identical specificity, the one defined later wins. But if one rule uses an ID selector and the other uses classes only, the ID rule wins even if it appears first in the file.

Why This Design Exists

The cascade is designed for predictability and maintainability. Allowing high-specificity selectors to override lower ones regardless of order prevents unexpected behavior in large projects. It encourages intentional styling rather than relying on file order.

Understanding this rule eliminates many debugging frustrations. When a style doesn’t apply, the issue is almost always specificity, not ordering.

Source order is the final step, not the first.
